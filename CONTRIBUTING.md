バグ報告の際は、以下の情報を含めてください：

- 期待される動作と実際の動作
- クラッシュログやスクリーンショット（可能な場合）

## ソースコードの構成

```text
AviQtl/
├── core/             # 基盤ロジック（デコーダ、設定管理、プロジェクト保存・読込）
├── engine/           # 信号処理・タイムライン計算（オーディオ、ECSシステム）
├── rendering/        # 描画エンジン実装（Google Filament 統合）
├── ui/               # ユーザーインターフェース
│   ├── include/      # C++ サービス・モデル定義
│   ├── src/          # C++ 実装
│   └── qml/          # QML/JS/GLSL 定義
│       ├── effects/  # エフェクト定義（.qml / .frag / .json）
│       ├── objects/  # 描画オブジェクト（動画 / 画像 / テキスト等）
│       └── common/   # 共通コンポーネント & 共通シェーダ
├── scripting/        # スクリプトエンジン (LuaJIT FFI)
├── i18n/             # 多言語翻訳ファイル (.ts)
├── assets/           # 静的リソース（アイコン、スプラッシュ）
├── vendor/           # 外部依存 SDK (Filament, Carla)
├── BUILD.py          # クロスプラットフォーム・ビルドスクリプト
├── check.py          # 静的解析・フォーマット一括実行
├── count.py          # コード行数集計ツール
└── format.fish       # コードフォーマッター
```

### 設計指針

- **データ指向型 (ECS):** 高いパフォーマンスとキャッシュ効率を優先します。
- **モダン C++:** C++23 の機能を活用し、安全で効率的なコードを目指します。
- **クロスプラットフォーム:** 特定の OS API への依存を避け、抽象化レイヤーを介するようにします。

### 技術スタック

| カテゴリ | 採用技術 |
|----------|----------|
| 言語 | C++23 (DOD/SoA 構造), QML, JavaScript, LuaJIT (FFI), GLSL |
| UI フレームワーク | Qt 6.11+ (Core, Gui, Quick, Multimedia, ShaderTools, 5Compat, Svg, Network, Concurrent) |
| 描画エンジン | Google Filament (PBR Engine), Vulkan / OpenGL / Metal 対応 |
| メディア処理 | FFmpeg (ハードウェアアクセラレーション対応), stb_image |
| 音声エンジン | Carla SDK (VST3 / LV2 / LADSPA / CLAP ホスト), Lilv, FFTW |
| スクリプト | LuaJIT (高性能なプラグイン・拡張ロジック) |
| ビルド & ツール | CMake, Ninja, Python 3, PySide6 (ビルドスクリプト), Clang (コンパイラ), Mold (リンカ) |

## 開発の進め方

### 1. 準備

- [Codeberg のリポジトリ](https://codeberg.org/taisho-guy/AviQtl)をフォークします。
- フォークしたリポジトリをローカルにクローンします。

### 2. 環境構築

- `README.md` の「ビルド手順」に従い、依存関係をインストールします。
- `BUILD.py` を使ってビルドが通ることを確認してください。

### 3. イシューの作成

- 作業を開始する前に、必ず [Issues](https://codeberg.org/taisho-guy/AviQtl/issues) で同様の報告や提案がないか確認してください。
- **提案・相談:** 新機能の実装や大規模なリファクタリングを行う場合は、実装前にイシューを作成し、設計方針（特に ECS 周りやアーキテクチャの責務分離）についてメンテナーと合意形成することを強く推奨します。
- **バグ報告:** 前述の「良いイシューを書くために」を参考に、再現環境や手順を詳しく記載してください。

### 4. ブランチの作成

変更内容に合わせて適切な名称のブランチを作成してください。

```bash
git switch -c fix/bug-description   # バグ修正
git switch -c feat/feature-name     # 新機能追加
```

### 5. コミットとプッシュ

- 変更内容が伝わるコミットメッセージを記述してください。
- 変更を自分のフォークへプッシュします。

### 6. プルリクエストの作成

- Codeberg 上でオリジナルリポジトリに対して PR を作成します。
- PR の説明には、変更の理由と修正内容を簡潔に記載してください。
